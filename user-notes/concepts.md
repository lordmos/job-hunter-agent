# 核心概念总结 — 第一阶段

## 1. LLM 能做什么 / 不能做什么

| 能做 | 不能做 |
|------|--------|
| 语义理解和推理（Node.js → REST API） | 执行代码、调外部 API |
| 按格式生成文本 | 记住上下文（无状态） |
| 从自然语言提取结构化信息 | 实时获取信息（没有联网） |
| 做判断和规划 | 保证输出 100% 准确 |

## 2. Tool Calling（Function Calling）

**用途：** 补齐 LLM 的短板——让 LLM 能触发"具体的事情"（查数据库、调 API、执行代码）。

**流程（2次 LLM 调用）：**
```
第1次调用：发消息 + 工具定义 → LLM 返回 tool_call（不执行，只返回参数）
Agent 执行工具函数
第2次调用：原消息 + tool_call + 工具结果 → LLM 给用户最终回复
```

**关键：** LLM 不执行函数，只决定调用哪个函数、传什么参数。执行是 Agent 侧的事。

## 3. Prompt Engineering + 结构化输出

**两种控制输出格式的手段，必须配合使用：**

| 手段 | 作用 | 单独使用的问题 |
|------|------|--------------|
| prompt 里描述 JSON schema | 告诉 LLM 输出哪些字段 | LLM 可能加 markdown 包装、加解释文字 |
| `response_format=json_object` | 保证输出语法合法的 JSON | 字段名由 LLM 自由发挥，不可控 |

**结论：两个都要用。** `response_format` 保证语法，prompt 里的 schema 保证字段结构。

## 4. "带工具的 LLM 应用" vs "Agent"

当前实现是前者——编排逻辑写死在代码里，LLM 只是执行每一步。

真正的 Agent：LLM 自主规划步骤顺序、决定工具调用时机、判断任务何时完成（ReAct loop）。第二阶段用 LangGraph 实现。

## 5. 短期 Memory 原理

LLM 无状态，每次调用都是全新请求。"记住上下文" = 把历史消息拼进 `messages` 数组重新发送。代价：消息越长，token 消耗越多，有上下文窗口上限。
